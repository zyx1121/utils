# rca — Root Cause Analysis (5-Why)

修了不只是讓現象消失，是讓「會產生這個現象的根因」消失。差別在下次同樣的坑會不會再炸一次。

## Trigger 信號

- 同一個 bug 改 2 次以上還沒掉
- "why isn't this working" / "為什麼還不行" / "怎麼又錯了"
- test 紅了改完還紅
- 線上異常重複出現
- 修法看起來太巧（"加個 try/except 就好了"）通常是症狀解，不是根因解

## Procedure

### Step 1 — 精確描述症狀

寫一行：什麼動作、什麼預期、實際是什麼。不寫「壞掉」「不 work」這種模糊話。

### Step 2-6 — 連問 5 個 Why，每個 Why 附證據

每個 Why 的答案**必須有可驗證來源**：

- 錯誤訊息原文（pasted）
- 程式碼位置（`file.py:42`）
- log 行
- 復現實驗結果
- 官方文件 / spec 連結
- git blame / 歷史 commit

格式：

```
[RCA-1/5] Why: <現象問題> → <答案> (證據: <來源>)
[RCA-2/5] Why: <上一答案問題> → <答案> (證據: ...)
...
```

5 不是硬規則，停在「再往下問就是無關因素或循環」那一層。

### Step 7 — 識別 root cause

最深的非循環 Why 就是根因。標 `[RCA-root: <一句話>]`。

### Step 8 — 修根因，不修症狀

如果修法是「在症狀那層擋一下」（加 try/except、加 retry、加 if-check）— 那是繃帶。明確標出來，並回頭問：根因可不可以修？不能修是因為什麼限制？

### Step 9 — 同類問題泛化

修完問：

- 這個根因在**同模組**還有沒有產生其他 bug？
- 上下游有沒有受影響？
- 同型態的錯誤在 codebase 別處有沒有？

修一個點就收工 = 頭痛醫頭。修完不泛化，等下次同樣坑再炸一次。

## Anti-pattern

- 「應該是環境問題」「API 不支援吧」— 沒驗證就歸因 = 甩鍋，違反三紅線 fact-driven
- 證據是「我覺得」「之前看過」— 不算證據，去找文件 / 跑指令確認
- 修完不驗證根因消失 — 違反 close loop

## Marker

`[RCA-N/5: Why ... → Answer ... (Evidence: ...)]`
`[RCA-root: ...]`
`[RCA-generalized: 同類問題 / 無同類]`

## 失敗時切換

- 5-Why 撞到「我不知道」且找不到證據 → 切 **tidy**（先清環境、確保跑的是預期版本）→ 切 **cunningham**（寫最小複現）
- 5-Why 答案開始循環 → 證據還不夠，回 source-first 補資料
